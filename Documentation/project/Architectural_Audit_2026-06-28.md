### MODULE 1 — Autenticación y Usuarios (`auth/`)

#### 1. Is the current implementation compatible with the real database?
**Partially.** The `usuario` entity maps to the `usuario` table with mostly correct columns (`id`, `username`/email, `password`, `nombre`, `rol_id`, `activo`, `creado_en`). The FK to `rol` is stored as a plain `int` (`rol_id`) without a JPA `@ManyToOne` relationship. There is no `Rol` entity, so the foreign key is orphaned in the database — `ddl-auto=update` never creates the `rol` table.

#### 2. Which entities are missing?
`Rol` entity mapping to table `rol` (id, nombre, descripcion). Without it, `usuario.rol_id` references a non-existent table.

#### 3. Which entities should not exist?
None. `User` is correctly scoped to this module.

#### 4. Which business rules were simplified?
**CRITICAL:** `UserDetailsServiceImpl` grants `Collections.emptyList()` as authorities. The `rolId` is never loaded into the Spring Security context, so RBAC is structurally dead. `UserService.createUser()` hardcodes `rolId = 1` and `nombre = email`, preventing proper role assignment. The use case diagram defines 4 roles with strictly different permissions (RF_21); none are enforced.

#### 5. Which REST endpoints are missing?
- `GET /api/roles` — list roles for user creation form
- `PUT /api/users/{id}/rol` — change user role (Administrador only)
- `PUT /api/users/{id}/activo` — activate/deactivate user

#### 6. Which services should be rewritten?
`UserService.createUser()` must accept `nombre` and `rolId` from the caller, not hardcode them. `UserDetailsServiceImpl.loadUserByUsername()` must load the role name and construct a `GrantedAuthority`.

#### 7. Which DTOs become obsolete?
`CreateUserRequest` must add `nombre` and `rolId` fields. `UserResponse` should include `rolNombre` for display.

#### 8. Which repositories should disappear?
None.

#### 9. Which repositories should be added?
`RolRepository extends JpaRepository<Rol, Integer>` for seed validation and role lookup.

#### 10. Which controllers need redesign?
`UserController` must add role management endpoints. `AuthController` must return role information in `LoginResponse` so the frontend can enforce role-based navigation.

#### 11. Which frontend views become obsolete?
None. `LoginView` is correct.

#### 12. Which frontend services become obsolete?
None. `authService.ts` is correct in structure.

#### 13. Which authentication/authorization rules are incomplete?
**All of them.** `SecurityConfig` grants `anyRequest().authenticated()` which means every authenticated user can access every endpoint regardless of role. RF_21 requires that only Recepcionista/Auxiliar can write reservations, only Contador can manage anticipos/facturas/penalidades, and only Administrador can configure policies and manage users.

#### 14. Which audit logging requirements are missing?
User creation and role changes must be recorded in `log_transaccion` (RF_18). Currently no audit log exists.

#### 15. Which financial workflow requirements are missing?
N/A for this module directly.

#### 16. Which automatic processes are missing?
None for this module.

#### 17. Which entities should reference each other differently?
`User` should have a `@ManyToOne Rol rol` relationship, not a plain `int rolId`.

#### 18. Which calculations are incomplete?
N/A.

#### 19. Which tables of the real database are completely ignored?
`rol` table — no entity, no seed data, no FK enforcement.

#### 20. Which project requirements are satisfied and which are not?
- RF_21 (RBAC): **NOT SATISFIED** — all roles have identical access
- RNF_01 (transactional integrity): **SATISFIED** — login is read-only, safe
- RF_18 (audit trail for user operations): **NOT SATISFIED**

**Severity: HIGH**

### MODULE 2 — Propiedad (`property/`)

#### 1. Is the current implementation compatible with the real database?
**No.** The `Property` entity maps to `propiedad` but with completely different columns. The original schema has: `id, nombre, direccion, descripcion, activa`. The implementation has: `id, name, address, city, capacity, pricePerNight` and is missing `descripcion` and `activa`.

#### 2. Which entities are missing?
No missing entities within `propiedad` itself, but the absence of `politica_cancelacion` (which has `propiedad_id` as FK) means `propiedad` cannot function as a root for policy configuration.

#### 3. Which entities should not exist?
The three invented columns (`city`, `capacity`, `pricePerNight`) should not exist on `propiedad`. The original system stores price information in `reserva.monto_total` (negotiated per booking), not on the property itself.

#### 4. Which business rules were simplified?
**CRITICAL:** `activa` soft-delete flag is absent. RF_03 requires that only active properties can receive reservations. CU_01 precondition: "La propiedad que se va a reservar debe estar registrada y activa en el sistema." This precondition cannot be enforced without `activa`.

**HIGH:** Price-per-night on the property implies a fixed rate per night, which contradicts the original model where `monto_total` is negotiated at booking time (which may vary by channel, season, special deals). The current `InvoiceService` calculates `subtotal = pricePerNight × nights`, which is wrong — RF_16 requires `base = monto_total − descuento_anticipo`.

#### 5. Which REST endpoints are missing?
- `PUT /api/propiedades/{id}/activar` — reactivate a property
- `PUT /api/propiedades/{id}/desactivar` — soft-delete
- `GET /api/propiedades/{id}/politicas` — list cancellation policies for a property

#### 6. Which services should be rewritten?
`PropertyService.delete()` should perform soft-delete (`activa = false`), not hard-delete. A check for active reservations before deactivation should be added.

#### 7. Which DTOs become obsolete?
`CreatePropertyRequest` and `PropertyResponse` must remove `city`, `capacity`, `pricePerNight` and add `descripcion`, `activa`.

#### 8. Which repositories should disappear?
None.

#### 9. Which repositories should be added?
None (for this module).

#### 10. Which controllers need redesign?
`PropertyController.delete()` must redirect to a `PUT .../desactivar` pattern with 204; hard DELETE should be rejected if the property has historical reservations.

#### 11. Which frontend views become obsolete?
`PropertiesView.vue` must be rewritten — remove capacity/city/price fields, add descripcion, add activa toggle.

#### 12. Which frontend services become obsolete?
`propertyService.ts` request/response DTOs must change completely.

#### 13. Which authentication/authorization rules are incomplete?
Creating and deactivating properties should be restricted to Administrador (RF_21). Currently no role check exists.

#### 14. Which audit logging requirements are missing?
Property creation and deactivation must be logged in `log_transaccion` (RF_18).

#### 15. Which financial workflow requirements are missing?
**HIGH:** `pricePerNight` is the wrong financial concept. `reserva.monto_total` is the correct field for financial calculation. The entire invoice calculation chain depends on this.

#### 16. Which automatic processes are missing?
Season auto-assignment on reservation (RF_05) requires seasons to exist with date ranges — which in turn requires `propiedad` to have correct configuration.

#### 17. Which entities should reference each other differently?
`propiedad` is referenced by `politica_cancelacion.propiedad_id` (not implemented), `reserva.propiedad_id` (partially correct — FK exists but wrong schema).

#### 18. Which calculations are incomplete?
Price-per-night on the entity is architecturally wrong for the invoice calculation model.

#### 19. Which tables of the real database are completely ignored?
`politica_cancelacion` (FK references `propiedad_id`) — entirely absent.

#### 20. Which requirements are satisfied and which are not?
- RF_03 (activa flag): **NOT SATISFIED**
- Basic CRUD: **PARTIALLY SATISFIED** (wrong schema)
- CU_01 precondition (activa check): **NOT SATISFIED**

**Severity: HIGH**

### MODULE 4 — Temporada (`temporada/`)

#### 1. Is the current implementation compatible with the real database?
**No implementation exists.** `temporada` defines named seasons with date ranges. `reserva.temporada_id` is nullable — a reservation can be created without a matching season if none is configured. However CU_01 precondition states: "Debe existir al menos una temporada configurada que cubra las fechas de la reserva."

#### 4. Which business rules were simplified?
**CRITICAL:** Season auto-assignment (RF_05, CU_01 step 6) — the system is supposed to automatically identify and assign the season covering the reservation dates. Currently no such logic exists. If dates cross two seasons, the system applies the majority-duration one or requests manual confirmation (per CU_01 notes). This is complex logic requiring a season range query.

#### 5. Which REST endpoints are missing?
- `GET /api/temporadas` — list seasons
- `POST /api/temporadas` — configure a season (Administrador)
- `GET /api/temporadas/vigente?checkIn=&checkOut=` — internal endpoint for auto-assignment

#### 20. Summary
Temporada is mandatory for CU_01 precondition. Its absence means the reservation form cannot auto-assign a season, violating RF_01 and RF_05.

**Severity: HIGH**

### MODULE 5 — Reserva (root aggregate) (`reservation/`)

This is the most critical module. Every financial workflow depends on this being correct.

#### 1. Is the current implementation compatible with the real database?
**No.** The implementation diverges on every structural dimension:

| Field | Schema | Implementation | Status |
|-------|--------|----------------|--------|
| `canal_id` | NOT NULL FK | Absent | MISSING |
| `temporada_id` | Nullable FK | Absent | MISSING |
| `politica_cancelacion_id` | NOT NULL FK | Absent | MISSING |
| `usuario_creador_id` | NOT NULL FK | Absent | MISSING |
| `monto_total` | NOT NULL decimal | Absent | MISSING |
| `no_show` | boolean | Absent | MISSING |
| `modificado_en` | timestamp | Absent | MISSING |
| `huesped_id` | Not in schema | Present | INVENTED |
| `cantidad_huespedes` | Not in schema | Present | INVENTED |
| Estado values | pendiente/confirmada/cancelada/completada/no_show | CONFIRMED/CANCELLED/COMPLETED | WRONG |

#### 2. Which entities are missing?
`Canal`, `Temporada`, `PoliticaCancelacion` (all missing; all required as FKs on this entity).

#### 3. Which entities should not exist?
`Guest (huesped)` as a separate JPA entity should not exist. The CU_01 form shows "Huésped" as a free-text/search field ("Nombre o documento") with no FK to a separate table. The `Esquema_BD.sql` has no `huesped` table. The entire `Guest` module was invented and must be removed.

#### 4. Which business rules were simplified?
**CRITICAL — RF_02:** A reservation cannot reach "Confirmada" state without a cancellation policy linked. CU_02 defines a two-step flow: (1) create reservation → status "Activa/Pendiente"; (2) link policy → status "Confirmada". Currently there is no policy field, so this transition is impossible. Any reservation is immediately CONFIRMED on creation, violating the mandatory policy requirement.

**CRITICAL — RF_01/RF_05:** Season auto-assignment is absent. The system should auto-detect which season covers the booking dates.

**CRITICAL — RF_06:** Reservation number must be auto-generated and unique (format implied: RES-{YEAR}-{SEQ}). CU_02 mockup shows "Reserva #RES-0042". Currently the reservation has no such generated number.

**CRITICAL — RF_18:** The creating user (`usuario_creador_id`) must be recorded. CU_01 postcondition: "El historial de la reserva contiene la entrada inicial con usuario, fecha y hora de creación." Currently there is no user reference on the reservation.

**HIGH — monto_total:** The negotiated total amount is absent. This field is the financial anchor of the entire system — it feeds the invoice calculation. Without it, RF_16 cannot be implemented.

**HIGH — no_show flag:** Missing. The status value `no_show` in the schema implies that no-show is a distinct terminal state, not just a sub-case of cancellation.

#### 5. Which REST endpoints are missing?
- `PUT /api/reservas/{id}/politica` — link cancellation policy (CU_02)
- `PUT /api/reservas/{id}/confirmar` — transition pendiente → confirmada (post policy assignment)
- `PUT /api/reservas/{id}/completar` — mark as completed
- `PUT /api/reservas/{id}/cancelar` — cancel (triggers penalty calculation)
- `PUT /api/reservas/{id}/no-show` — mark as no-show
- `GET /api/reservas/{id}/historial` — change history (RF_18)
- `GET /api/reservas/{id}/trazabilidad` — full financial breakdown

#### 6. Which services should be rewritten?
`ReservationService` must be completely rewritten:
- Add season auto-assignment logic
- Enforce `politica_cancelacion_id` before confirming
- Implement FSM: pendiente → confirmada → completada/cancelada/no_show
- Write to `historial_reserva` on every state change (RF_18)
- Record `usuario_creador_id` from `SecurityContextHolder`
- Enforce property `activa` check (RF_03)
- Enforce no-overlap only for CONFIRMED reservations

#### 7. Which DTOs become obsolete?
`CreateReservationRequest` — entire DTO invalid. Current fields: `propertyId, guestId, checkIn, checkOut, guestCount`. Required fields: `propiedadId, canalId, temporadaId (optional), checkIn, checkOut, huespedNombre, huespedDocumento, monto_total, notas`.
`ReservationResponse` — all field names and structure invalid.

#### 8. Which repositories should disappear?
`GuestRepository` — after reservation is corrected, the guest FK reference disappears. The guest module as a whole should be removed.

#### 9. Which repositories should be added?
`HistorialReservaRepository` for writing audit trail entries.

#### 10. Which controllers need redesign?
`ReservationController` — must split CRUD into state-machine endpoints. Current `PUT /api/reservations/{id}` (generic update) should be replaced by explicit state transitions.

#### 11. Which frontend views become obsolete?
`ReservationsView.vue` must be completely rewritten. The creation form must include: Propiedad dropdown, Canal de origen dropdown, Huésped text field, Temporada (auto-populated read-only), Monto total, Check-in, Check-out, Notas. The CU_01 mockup is the specification.

#### 12. Which frontend services become obsolete?
`reservationService.ts` — all types and API calls must change.
`types/reservation.ts` — `ReservationStatus` enum values are wrong.

#### 13. Which authentication/authorization rules are incomplete?
Per RF_21 and the use case diagram: only Recepcionista and Auxiliar contable can create reservations. Administrador can also. Contador cannot. Currently no role check exists.

#### 14. Which audit logging requirements are missing?
Every state change must write to `historial_reserva` with: campo changed, valor anterior, valor nuevo, usuario, fecha, hora (RF_18, CU_02 postconditions). This is entirely absent.

#### 15. Which financial workflow requirements are missing?
`monto_total` — the negotiated booking value. Without it, the entire invoice chain (RF_16) cannot be computed correctly. Currently the system approximates with `pricePerNight × nights` which is architecturally wrong.

#### 16. Which automatic processes are missing?
- Season auto-assignment (RF_05)
- Reservation number auto-generation
- `historial_reserva` write on creation (RF_18, CU_01 postcondition: "El historial de la reserva contiene la entrada inicial")

#### 17. Which entities should reference each other differently?
- `reserva` → `propiedad` (via propiedadId): EXISTS but property has wrong schema
- `reserva` → `canal` (via canalId): MISSING — must be NOT NULL FK
- `reserva` → `temporada` (via temporadaId): MISSING — nullable FK
- `reserva` → `politica_cancelacion` (via politicaCancelacionId): MISSING — must be NOT NULL before confirming

#### 18. Which calculations are incomplete?
Availability check exists (overlap detection in `ReservationRepository`) but only checks CONFIRMED reservations — this is actually correct per the use case (pendiente reservations don't block availability). However the status enum mismatch (`CONFIRMED` vs `confirmada`) means the query would need to be updated when status values are corrected.

#### 19. Which tables of the real database are completely ignored?
`historial_reserva` — entire audit trail table missing.

#### 20. Which requirements are satisfied and which are not?
- RF_01 (canal, temporada, trazabilidad): **NOT SATISFIED**
- RF_02 (mandatory policy): **NOT SATISFIED**
- RF_03 (activa check): **NOT SATISFIED**
- RF_04 (season precondition): **NOT SATISFIED**
- RF_05 (auto-assign season): **NOT SATISFIED**
- RF_06 (unique reservation number): **NOT SATISFIED**
- RF_18 (audit trail): **NOT SATISFIED**
- RNF_01 (transactional integrity): **PARTIALLY** — @Transactional exists in some services
- Basic availability overlap check: **SATISFIED** (correct logic, wrong enum values)

### MODULE 8 — Anticipo (Advance Payment) (`anticipo/`)

#### 1. Is the current implementation compatible with the real database?
**No implementation exists.** The current system has an invented `pago` (payment) entity that serves a completely different purpose. `anticipo` in the original schema is: `id, reserva_id, usuario_id, monto, fecha_pago, metodo_pago, estado (registrado/aplicado/devuelto)`.

#### 3. Which entities should not exist?
`Payment (pago)` — the entire `payment/` package is an invention. `Payment` records a post-invoice settlement. `Anticipo` records a pre-stay advance. These are not equivalent:
- `pago.factura_id` → links to invoice (post-billing)
- `anticipo.reserva_id` → links to reservation (pre-billing)
- `pago` has no state machine
- `anticipo` has states: registrado → aplicado → devuelto
- `pago` is never fed into the invoice calculation
- `anticipo.monto` feeds `factura.descuento_anticipo`, which determines the invoice base

#### 4. Which business rules were simplified?
**CRITICAL — RF_09:** Registration of advances with status tracking is entirely absent.
**CRITICAL — RF_10:** Status FSM (registrado → aplicado → devuelto) is absent.
**CRITICAL — RF_11:** Advances as accounting liabilities (reflected in balance) requires the aplicado state and a query summing applied anticipos per reservation.
**CRITICAL — RF_16:** Invoice calculation requires `descuento_anticipo = Σ(anticipo.monto WHERE estado='aplicado')`. Without anticipos, the invoice always charges the full amount with no advance discount.

#### 5. Which REST endpoints are missing?
- `GET /api/reservas/{id}/anticipos` — list advances for a reservation
- `POST /api/reservas/{id}/anticipos` — register an advance (Contador)
- `PUT /api/anticipos/{id}/aplicar` — apply advance to reservation (Contador)
- `PUT /api/anticipos/{id}/devolver` — mark as returned

#### 8. Which repositories should disappear?
`PaymentRepository` — the entire payment module must be removed and replaced with `AnticipoRepository`.

#### 9. Which repositories should be added?
`AnticipoRepository` with custom query: `Σ(monto WHERE reserva_id = ? AND estado = 'aplicado')` for invoice discount calculation.

#### 13. Which authentication/authorization rules are incomplete?
Per use case diagram: only `Contador` can register and apply advances. Currently no role enforcement exists.

#### 15. Which financial workflow requirements are missing?
This module IS the financial workflow trigger. Without it, invoices cannot be correctly calculated (no `descuento_anticipo`), automatic refund obligations cannot be generated (CU_03 requires Σ anticipos > valor neto), and the RF_11 balance requirement is unmet.

#### 16. Which automatic processes are missing?

### MODULE 10 — Factura (Invoice) (`invoice/`)

#### 1. Is the current implementation compatible with the real database?
**No.** The `factura` table schema: `id, reserva_id, usuario_id, numero_factura (UNIQUE NOT NULL), subtotal, descuento_anticipo, impuestos, total, estado (borrador/emitida/anulada), url_documento`. The implementation has: `id, reservationId, subtotal, tax (wrong name), total, status (PENDING/PAID/CANCELLED — wrong values), version, createdAt`. Missing: `usuario_id`, `numero_factura`, `descuento_anticipo`, `url_documento`. Wrong: status values, column name `iva` vs `impuestos`. Extra: `@Version` (not in schema), `UNIQUE` constraint on `reservationId` (wrong — original allows multiple invoices per reservation cycle).

#### 4. Which business rules were simplified?
**CRITICAL — RF_14:** `numero_factura` is a required unique identifier per invoice. CU_02 mockup shows "Reserva #RES-0042" — the equivalent for invoices would be a similar sequential format. Without this, invoices have no legal identifier.

**CRITICAL — RF_16:** The invoice calculation is wrong:
- Current: `subtotal = pricePerNight × nights; tax = subtotal × 0.19; total = subtotal + tax`
- Required: `subtotal = reserva.monto_total; descuento_anticipo = Σ(anticipo aplicado); base = subtotal − descuento_anticipo; impuestos = base × 0.19; total = base + impuestos`
- The current formula ignores advance payment discounts entirely.

**CRITICAL — RF_15:** Invoice states are wrong. `PENDING/PAID/CANCELLED` maps to nothing in the original design. The original has `borrador/emitida/anulada`:
- `borrador` = draft (can be edited before emission)
- `emitida` = emitted (immutable, legally binding document)
- `anulada` = cancelled/annulled
- `PAID` is not an invoice state in the original — payment is handled via `anticipo`

The concept of "paying an invoice" via a `pago` record is entirely absent from the original design. Invoices don't have a "paid" state — they are either emitted (value collected via anticipos) or annulled.

**HIGH:** `usuario_id` (NOT NULL in schema) — the Contador who emits the invoice must be recorded (RF_18, RF_21).

**HIGH:** `url_documento` — the invoice PDF document link is missing.

**HIGH:** The `UNIQUE` constraint on `reservationId` is wrong. The original schema allows: (1) borrador → emitida; (2) anulada + nota_credito; (3) potentially revised invoice. A hard unique constraint blocks this workflow.

#### 5. Which REST endpoints are missing?

- `PUT /api/facturas/{id}/emitir` — emit invoice (Contador)
- `PUT /api/facturas/{id}/anular` — annul invoice
- `GET /api/facturas/{id}/nota-credito` — get associated credit note
- `POST /api/facturas/{id}/nota-credito` — issue credit note

#### 12. Which frontend services become obsolete?
`invoiceService.ts` — all types and status values must change. `types/invoice.ts` `InvoiceStatus` enum is wrong.

**Severity: CRITICAL**
- Effort: 2–3 days (after anticipo module exists)
- Requires migration: Yes (`ALTER TABLE factura` to add/rename/remove columns; drop unique constraint)
- Requires rewriting: Yes — calculation formula and state machine are wrong

## Section F — Recommended Migration Strategy

### Decision: OPTION A — Incrementally refactor current code

**Rationale:**

**Do NOT choose OPTION C (Rewrite backend and frontend) or OPTION D (Start over).**

The project has substantial reusable infrastructure that a complete rewrite would discard:
- Spring Boot 3.5 + JWT security stack — correct and functional
- `GlobalExceptionHandler` — correct pattern
- Feature-package structure — scales correctly to new modules
- Vue 3 + `useAsyncState` composable — reusable for all new views
- `AppLayout.vue`, `AppHeader.vue`, `AppNav.vue` — reusable
- `shared.css` — reusable across all future views
- Axios service layer pattern — correct
- `@Transactional` presence in new modules — correct pattern established

**Do NOT choose OPTION B (Rewrite only backend):**

The frontend views for Guest and Payment must be removed. The frontend views for Reservation, Invoice, and all missing modules must be rebuilt. This is frontend rewriting, not just backend rewriting. However, the rewrite is targeted (remove 2 views, rebuild 3, add 8) — not a complete frontend rewrite.

**Why OPTION A — Incremental refactor:**

The scope of changes is large but bounded. It can be parallelized across sprints without halting current development. The correct execution order is:

1. **Sprint 0 (1 day):** Adopt Flyway (`V1__initial_schema.sql` = `Esquema_BD.sql` verbatim); set `ddl-auto=validate`; establish schema baseline
2. **Sprint 1 (3 days):** Implement `Rol`, `Canal`, `Temporada`, `PoliticaCancelacion`, `ReglasPenalidad` — reference data modules with no structural dependencies
3. **Sprint 2 (5 days):** Rebuild `Reserva` root aggregate — add all missing FKs, `monto_total`, `no_show`, correct status enum, state machine, `historial_reserva` writes, reservation number generation; remove `Guest` module
4. **Sprint 3 (4 days):** Implement `Anticipo` module (replace `Payment`); implement `Penalidad` calculation engine; implement `Devolución` with automatic trigger on invoice emission
5. **Sprint 4 (3 days):** Rebuild `Factura` — correct calculation formula with `descuento_anticipo`, sequential `numero_factura`, correct states, `usuario_id`; implement `NotaCredito`
6. **Sprint 5 (2 days):** Implement `LogTransaccion` audit writes across all services; implement RBAC enforcement in `SecurityConfig`
7. **Sprint 6 (2 days):** Rebuild frontend — remove GuestsView/PaymentsView; rebuild ReservationsView/InvoicesView; add PenalidadesView/AnticiposView; add refund badge indicator

**What must be removed (not refactored):**
- `Guest` entity + all layers (both backend and frontend)
- `Payment` entity + all layers (both backend and frontend)
- `DevelopmentDataSeeder` will need to be rewritten in Sprint 3 once the correct entities exist

**What must not be touched:**
- `auth/` module — correct; only add RBAC wiring
- `SecurityConfig` — correct; only add route-level authorization rules
- All frontend infrastructure files (AppLayout, AppHeader, useAsyncState, shared.css, router infrastructure)
- `booking/` package POJOs — keep as specification; wire `BookingValidator` as `@Service` in Sprint 2; wire `InvoiceCalculator` logic into correct `InvoiceService` in Sprint 4

**Risk assessment:**

The highest risk is Sprint 2 (Reserva rebuild) because it is the root aggregate. Every subsequent module (anticipo, penalidad, factura, devolucion) depends on the reservation being correct before it can be implemented. If Sprint 2 is delayed or incorrectly executed, all subsequent sprints are blocked. Sprint 2 should be reviewed by all team members before starting Sprint 3.

The `DevelopmentDataSeeder` will produce incorrect data until Sprint 3 is complete (it seeds Guest and Payment entities that will not exist after the rebuild). It should be disabled (`@Profile("never")`) at the start of Sprint 1 and re-enabled with correct data generation in Sprint 3.
