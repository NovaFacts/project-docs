## PILLAR 1 — Schema Coherence (Database vs Backend)

### CRITICAL-1: Integer/BigInt FK Type Mismatch in Flyway V5 and V6

**Severity:** CRITICAL — would cause `V5`/`V6` Flyway migrations to fail on a fresh PostgreSQL database.

The `reserva` table was created in V1 as:
```sql
id BIGSERIAL PRIMARY KEY   -- PostgreSQL BIGINT (8 bytes)
```

Flyway V5 then creates `anticipo` and `penalidad` with:
```sql
reserva_id INT NOT NULL,   -- INTEGER (4 bytes)
usuario_id INT NOT NULL,   -- INTEGER (4 bytes)
FOREIGN KEY (reserva_id) REFERENCES reserva(id),
FOREIGN KEY (usuario_id) REFERENCES usuario(id)
```

Similarly, Flyway V6 creates `factura`, `nota_credito`, and `devolucion` with:
```sql
reserva_id INT NOT NULL,
usuario_id INT NOT NULL,
```

PostgreSQL **rejects** a FK constraint from `INTEGER (int4)` to `BIGINT (int8)` with:
```
ERROR: foreign key constraint cannot be implemented
DETAIL: Key columns "reserva_id" and "id" are of incompatible types: integer and bigint.
```

**Why tests pass:** The test profile uses H2 with `spring.jpa.hibernate.ddl-auto=create-drop` and `spring.flyway.enabled=false`. H2 derives the schema from Java entities, which use `Long` (→ BIGINT correctly). The Flyway scripts are **never executed in any test**.

**Affected tables:** `anticipo`, `penalidad`, `factura`, `nota_credito`, `devolucion` (all V5+V6 additions)

**Must-Fix:** Change all `INT` PK/FK declarations in V5 and V6 to `BIGINT`:
```sql
-- V5:
id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY
reserva_id BIGINT NOT NULL
usuario_id BIGINT NOT NULL
-- V6: same pattern for factura, nota_credito, devolucion
```

### HIGH-1: `propiedad_id` in `reserva` — Manual ID Mapping, No DB FK Constraint

**Severity:** HIGH — violates the "no manual ID mapping" requirement; breaks referential integrity.

The `Reservation` entity maps `propiedad_id` as a plain column, not a `@ManyToOne`:
```java
// reservation/entity/Reservation.java
@Column(name = "propiedad_id", nullable = false)
private Long propertyId;    // bare Long, NOT @ManyToOne Property
```

Neither V1 (which adds the column) nor any subsequent migration adds a FK constraint on `reserva.propiedad_id → propiedad.id`. The uniqueness/existence check is delegated to application code in `ReservationService`:
```java
if (!propertyRepository.existsById(request.getPropertyId())) {
    throw new ResponseStatusException(HttpStatus.NOT_FOUND, "Propiedad no encontrada");
}
```

**Consequences:**
1. A reservation can reference a deleted (soft-deleted) `propiedad` — the DB won't catch it.
2. A reservation can be updated to point to a non-existent `propiedad_id` if the validation is bypassed.
3. The entity graph is incomplete: `Reservation` cannot navigate to its `Property`.

**Must-Fix:** Add `@ManyToOne(fetch = FetchType.LAZY) @JoinColumn(name = "propiedad_id", nullable = false) private Property property;` to the `Reservation` entity and create a Flyway V7 migration that adds the DB-level FK constraint.

---

### HIGH-2: `factura` — No DB-Level UNIQUE Constraint on `reserva_id`

**Severity:** HIGH — uniqueness enforced only in application code, vulnerable to race conditions.

The V6 migration does NOT declare `reserva_id UNIQUE` on the `factura` table. `FacturaService.create()` checks:
```java
if (facturaRepository.existsByReservaId(request.getReservaId())) {
    throw new ResponseStatusException(HttpStatus.CONFLICT, "Ya existe una factura para esta reserva");
}
```

Under concurrent requests, two threads can both pass the `existsByReservaId()` check (a classic check-then-act race), then both insert, resulting in two facturas for one reserva. The DB has no constraint to reject the second insert.

**Must-Fix in V7:** `ALTER TABLE factura ADD CONSTRAINT uq_factura_reserva UNIQUE (reserva_id);`

### MEDIUM-2: `anticipo.estado` and `devolucion.estado` — Unvalidated String State Machine

**Severity:** MEDIUM — no DB CHECK constraint and no DTO validation enforce valid state values.

The DB uses `VARCHAR(50)` for estado fields with no CHECK constraint. Service code uses raw string literals:
- Anticipo: "registrado", "aplicado", "devuelto"  
- Devolucion: "pendiente", "procesada", "rechazada"

No enum type exists for these states. A direct database INSERT or a future service bug could corrupt the state machine.

**Must-Fix:** Add enums (or at minimum DB CHECK constraints) and enum-to-string `@Enumerated` mappings.

### HIGH: Dashboard `totalGuests` — Semantic Field Name Mismatch

**Severity:** HIGH (semantic contract violation)

`DashboardService.getSummary()` passes:
```java
return new DashboardResponse(
    reservationRepository.count(),  // ← passed as totalGuests
    ...
```

The first constructor parameter is `totalGuests`, but the value is **the total number of reservations** (`reservationRepository.count()`), not a guest count. There is no guest count metric (the `huesped` table was dropped). The `DashboardView.vue` renders this as "Total Reservas" which is accurate at the view level, but the DTO field is misleading. If a future developer adds a guest count and puts it in `totalGuests`, they will replace this reservation count.

**Must-Fix:** Rename `totalGuests → totalReservations` in both `DashboardResponse.java` and `types/dashboard.ts`, and update `DashboardService` and `DashboardView.vue`.

### MEDIUM-1: Enum Case Mismatch — `DevolucionEstado` vs Backend String Literals

**Severity:** MEDIUM

Frontend `types/devolucion.ts` defines:
```typescript
export type DevolucionEstado = 'pendiente' | 'procesada' | 'rechazada'
```

Backend `DevolucionService` stores lowercase strings: `"pendiente"`, `"procesada"`, `"rechazada"`. These match ✓.

BUT `FacturaResponse.estado` serializes `InvoiceStatus` enum as `PENDING`, `PAID`, `CANCELLED` (uppercase). Frontend `FacturaEstado = 'PENDING' | 'PAID' | 'CANCELLED'` matches ✓.

The inconsistency is that `Devolucion.estado` uses untyped lowercase strings while `Factura.estado` uses a proper Java enum serialized to uppercase. This creates a dual convention — one entity uses a proper type-safe enum, another uses bare strings. No runtime failure today, but a maintenance trap.

## PILLAR 3 — Security & RBAC Flow

### HIGH-1: Financial Read Endpoints Are Publicly Accessible to All Authenticated Users

**Severity:** HIGH — a `Recepcionista` or any authenticated user can read all financial records via the API, even though the frontend gates these pages.

`SecurityConfig` correctly restricts **write** operations (POST/PUT/DELETE) on financial paths. But **read** operations (GET) fall through to `anyRequest().authenticated()`:

| Endpoint | Should be restricted to | Current SecurityConfig rule |
|---|---|---|
| `GET /api/anticipos/**` | ADMINISTRADOR, CONTADOR, AUXILIAR_CONTABLE | `anyRequest().authenticated()` ❌ |
| `GET /api/penalidades/**` | ADMINISTRADOR, CONTADOR, AUXILIAR_CONTABLE | `anyRequest().authenticated()` ❌ |
| `GET /api/facturas/**` | ADMINISTRADOR, CONTADOR | `anyRequest().authenticated()` ❌ |
| `GET /api/notas-credito/**` | ADMINISTRADOR, CONTADOR | `anyRequest().authenticated()` ❌ |
| `GET /api/devoluciones/**` | ADMINISTRADOR, CONTADOR | `anyRequest().authenticated()` ❌ |

A `Recepcionista` can call `GET /api/facturas` and receive all invoices with full financial detail. The frontend navigation guard (`requiresBilling`, `requiresFinancial`) only protects the UI — it does not protect the API.

**Must-Fix in SecurityConfig:**
```java
// Add GET restrictions for financial read endpoints
.requestMatchers(HttpMethod.GET, "/api/anticipos/**", "/api/penalidades/**")
    .hasAnyRole("ADMINISTRADOR", "CONTADOR", "AUXILIAR_CONTABLE")
.requestMatchers(HttpMethod.GET, "/api/facturas/**", "/api/notas-credito/**", "/api/devoluciones/**")
    .hasAnyRole("ADMINISTRADOR", "CONTADOR")
```

### MEDIUM-2: `DisabledException` from Inactive-User JWT Returns Generic 401

**Severity:** MEDIUM — degraded user experience, not a security flaw.

When a deactivated user's valid JWT is presented, `JwtAuthenticationFilter` calls `loadUserByUsername()`, which returns a `disabled=true` `UserDetails`. Spring Security throws `DisabledException`, which is caught silently in:
```java
} catch (Exception e) {
    SecurityContextHolder.clearContext();
}
```

The request then receives a generic `401 Unauthorized` from `HttpStatusEntryPoint` with no body. The frontend (`api.ts` Axios interceptor) has no special handling for "account disabled" — it would appear as a generic authentication failure. The `UserService.login()` endpoint does return `"Cuenta de usuario desactivada"` on login attempts. JWT flow does not surface this message.

### LOW: `ROLE_AUXILIAR_CONTABLE` Authority Construction is Correct but Fragile

`UserDetailsServiceImpl` builds the authority as:
```java
"ROLE_" + user.getRol().getNombre().toUpperCase().replace(" ", "_")
```

For "Auxiliar contable" → `ROLE_AUXILIAR_CONTABLE`. This matches `SecurityConfig`'s `hasAnyRole("AUXILIAR_CONTABLE")`. ✓

But this is fragile: renaming the role in the database (e.g., "Auxiliar Contable" with capital C) would break RBAC silently. The authority should be derived from a stable identifier (the role `id`), not the display name.

## PILLAR 4 — Transactional Integrity & Constraints

### HIGH: `ReservationService.delete()` — Unguarded Hard Delete with FK Children

**Severity:** HIGH — deleting a reservation with associated anticipos, penalidades, or facturas will produce a `DataIntegrityViolationException` (FK violation) turned into a generic `409 "Conflicto de datos"` with no explanation.

```java
// ReservationService.java
@Transactional
public void delete(Long id) {
    reservationRepository.delete(getOrThrow(id));  // NO state check, NO child check
}
```

Compare with `AnticipoService.delete()` which guards:
```java
if ("aplicado".equals(anticipo.getEstado())) {
    throw new ResponseStatusException(HttpStatus.CONFLICT,
        "No se puede eliminar un anticipo ya aplicado a una factura");
}
```

`ReservationService.delete()` has no equivalent protection. The correct behavior should be to prevent deletion of any reservation that has associated financial records.

**Must-Fix:**
```java
@Transactional
public void delete(Long id) {
    Reservation reservation = getOrThrow(id);
    if (anticipoRepository.findByReservaId(id).size() > 0) {
        throw new ResponseStatusException(HttpStatus.CONFLICT,
            "No se puede eliminar una reserva con anticipos registrados");
    }
    if (facturaRepository.existsByReservaId(id)) {
        throw new ResponseStatusException(HttpStatus.CONFLICT,
            "No se puede eliminar una reserva con factura asociada");
    }
    reservationRepository.delete(reservation);
}
```

### MEDIUM-1: `DevolucionService.create()` — No Guard on Anticipo Estado Before Creating Devolucion

**Severity:** MEDIUM — a devolucion can be created for an anticipo that is already "devuelto" or "aplicado", resulting in double-refund.

```java
// DevolucionService.create()
Anticipo anticipo = anticipoRepository.findById(request.getAnticipoId())...;
// ← No check on anticipo.getEstado()
devolucion.setEstado("pendiente");
anticipo.setEstado("devuelto");
anticipoRepository.save(anticipo);
```

If the anticipo is already "devuelto" (a previous devolucion was created), calling `create()` again sets it to "devuelto" again and creates a second devolucion for the same anticipo. This is a double-refund scenario.

**Must-Fix:**
```java
if (!"registrado".equals(anticipo.getEstado())) {
    throw new ResponseStatusException(HttpStatus.CONFLICT,
        "El anticipo ya fue devuelto o está aplicado a una factura");
}
```

### MEDIUM-2: `numero_factura` and `numero_nota` Generated from 8-char UUID Prefix — Collision Risk

**Severity:** MEDIUM — low probability but high impact; collision triggers a `DataIntegrityViolationException` mapped to "Conflicto de datos" with no retry guidance.

```java
// FacturaService
String numeroFactura = "FAC-" + UUID.randomUUID().toString()
        .replace("-", "").substring(0, 8).toUpperCase();
```

Taking only the first 8 hex characters yields 16^8 = 4,294,967,296 possible values. With birthday paradox, a collision becomes statistically likely after ~65,536 facturas. The DB UNIQUE constraint will catch it but the error surfaces as an opaque `409`.

**Must-Fix:** Use the full UUID (36 chars), or generate a sequential number from a database sequence.

### MEDIUM-3: `LoginRequest` Missing `@Email` Format Validation

**Severity:** MEDIUM — malformed emails pass backend validation and fail only at the database lookup.

```java
// LoginRequest.java
@NotBlank
private String email;   // ← no @Email constraint
```

`CreateUserRequest` correctly has `@Email`. A login attempt with `"notanemail"` passes `@Valid` and reaches `UserService.login()`, which does a `findByUsername("notanemail")` → 401 "Credenciales inválidas". No crash, but the missing validation leaks a database query per malformed login attempt.

### LOW-1: No Soft-Delete on Financial Entities — Intentional but Undocumented

`AnticipoService.delete()` checks estado before performing a hard delete. `FacturaService.delete()` checks `estado != PAID`. `DevolucionService.delete()` checks `estado == "pendiente"`. These are appropriate state guards. The design decision to use hard delete for financial records (while using soft-delete for Users and Properties) is coherent but not documented. An incoming developer might assume all deletes are soft.

### LOW-2: `@Transactional` on `UserService.getUsers()` Correctly Uses `readOnly = true`

All read-only service methods across modules use `@Transactional(readOnly = true)`. Write methods use `@Transactional` without `readOnly`. This is correct throughout. ✓
