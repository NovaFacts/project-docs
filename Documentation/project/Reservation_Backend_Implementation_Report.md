## Executive Summary

The Reservation backend module has been implemented following the exact architectural template established by the Guest and Property modules. The module exposes five REST endpoints under `/api/reservations`, all protected by the existing JWT authentication filter. Seven business rules are enforced at the service layer before any persistence occurs: guest existence, property existence, date ordering, maximum stay (30 nights), guest count positivity, property capacity check, and date overlap detection. No existing module was modified. The module compiles cleanly alongside all 32 pre-existing tests.

## Database Model

Hibernate generates the following DDL via `ddl-auto=update`:

```sql
CREATE TABLE reserva (
    id                  BIGSERIAL       PRIMARY KEY,
    huesped_id          BIGINT          NOT NULL,
    propiedad_id        BIGINT          NOT NULL,
    fecha_inicio        DATE            NOT NULL,
    fecha_fin           DATE            NOT NULL,
    cantidad_huespedes  INTEGER         NOT NULL,
    estado              VARCHAR(20)     NOT NULL,
    creado_en           TIMESTAMP       NOT NULL
);
```

**Notes:**
- `huesped_id` and `propiedad_id` are plain foreign key values stored as `BIGINT`. No `FOREIGN KEY` constraint is generated by Hibernate (no `@ManyToOne` is used), which is correct per the module specification.
- `estado` stores the enum constant name as a string (`CONFIRMED`, `CANCELLED`, `COMPLETED`). Using `EnumType.STRING` instead of `EnumType.ORDINAL` ensures that adding new constants in the future does not corrupt existing data.
- `fecha_inicio` and `fecha_fin` map to `LocalDate` → SQL `DATE` (no time component). This is intentional: check-in and check-out are calendar dates, not timestamps.
- `creado_en` is set once by `@PrePersist` and marked `updatable = false`.

**Relationship to `Esquema_BD.sql`:**  
The `reserva` table in the SQL schema includes additional fields (`monto_total`, `noches`, foreign key constraints). Those are out of scope for this implementation and will be added when the Invoice module is built.

## Business Rules Implemented

All rules are enforced inside `ReservationService` before any write to the database.

### 1. Guest must exist
`guestRepository.existsById(guestId)` — uses the default `JpaRepository` method.  
On failure: **404** `"Huésped no encontrado"`

### 2. Property must exist
`propertyRepository.findById(propertyId)` — must succeed to obtain the capacity.  
On failure: **404** `"Propiedad no encontrada"`

### 3. Date ordering
`checkIn.isBefore(checkOut)` must be true.  
On failure: **400** `"La fecha de inicio debe ser anterior a la fecha de fin"`  
Same-day reservations (checkIn == checkOut) are rejected by this check.

### 4. Maximum stay — 30 nights
`ChronoUnit.DAYS.between(checkIn, checkOut) <= 30`  
On failure: **400** `"La reserva no puede superar 30 noches"`

### 5. Guest count positivity
`@Positive` on `guestCount` in the DTO handles this at deserialization time.  
On failure: **400** via `MethodArgumentNotValidException`

### 6. Property capacity
`guestCount <= property.getCapacity()`  
On failure: **400** `"La cantidad de huéspedes supera la capacidad de la propiedad"`

### 7. Date overlap detection
Checked via a derived repository query. Two reservations on the same property overlap when:

```
existingCheckIn < newCheckOut  AND  existingCheckOut > newCheckIn
```

Touching boundaries are permitted:

```
Reservation A: [10 ──── 15]
Reservation B:            [15 ──── 20]   ← allowed, no conflict
```

On failure: **409** `"La propiedad ya tiene una reserva en esas fechas"`

For `update()`, the same reservation being updated is excluded from the check via `existsByPropertyIdAndCheckInBeforeAndCheckOutAfterAndIdNot(...)` — the same self-exclusion pattern used by `GuestRepository.existsByDocumentNumberAndIdNot` and `PropertyRepository.existsByNameIgnoreCaseAndIdNot`.

## Architectural Decisions

### 1. `ReservationStatus` placed in `entity/` package
The enum is used exclusively by the `Reservation` entity and its DTOs. The project has no dedicated `enums/` package. Placing it in `entity/` keeps it co-located with the type that owns it, consistent with how the project organizes domain types.

### 2. No JPA relationship annotations
`guestId` and `propertyId` are stored as plain `Long` columns with no `@ManyToOne`. Cross-module validation is performed by querying `GuestRepository` and `PropertyRepository` directly from `ReservationService`. This avoids tight coupling between modules, prevents accidental eager loading, and keeps the module boundary explicit.

### 3. `status` absent from `CreateReservationRequest`
New reservations are always `CONFIRMED` — this is a business invariant, not a user input. Exposing `status` in the create DTO would allow clients to create `CANCELLED` or `COMPLETED` reservations, which is semantically incorrect. The service hardcodes `ReservationStatus.CONFIRMED` on creation.

### 4. `status` present in `UpdateReservationRequest`
Updates must be able to transition a reservation to `CANCELLED` or `COMPLETED`. Including `status` in the update DTO with `@NotNull` is the minimal, explicit approach — consistent with how the project handles all other required update fields.

### 5. Overlap query uses Spring Data derived methods, not JPQL
The overlap check translates directly to Spring Data's `Before`/`After` keyword semantics:

```java
existsByPropertyIdAndCheckInBeforeAndCheckOutAfter(Long propertyId, LocalDate checkOut, LocalDate checkIn)
// → WHERE propiedad_id = ? AND fecha_inicio < ? AND fecha_fin > ?
```

No JPQL or native SQL was needed. This is consistent with how `GuestRepository` and `PropertyRepository` express their queries.

### 6. Controller GET methods return raw types
`getAll()` and `getById()` return raw `List<ReservationResponse>` and `ReservationResponse` respectively — consistent with `GuestController`. POST and DELETE use explicit `ResponseEntity` for the 201 and 204 status codes. PUT uses `ResponseEntity.ok()` for the explicit 200 — matching `PropertyController`.

### 7. No modification to `SecurityConfig`
`anyRequest().authenticated()` already covers `/api/reservations/**`. Adding a specific matcher would be redundant.

## Assumptions Made

1. **No minimum check-in date.** The spec does not require `checkIn >= today`. Reservations with past dates are accepted. This is a deliberate omission pending a business decision.

2. **Overlap check applies regardless of status.** A `CANCELLED` reservation is still counted in the overlap check. Removing cancelled reservations from the conflict window is a future improvement that requires a business decision about cancellation workflows.

3. **`guestId` and `propertyId` are not validated for positivity at the DTO level.** Any non-null `Long` is accepted; the service immediately performs an existence check and returns 404 if the referenced entity is not found. Negative IDs simply fail the existence check.

4. **`fecha_inicio` and `fecha_fin` use `DATE`, not `TIMESTAMP`.** Check-in/out are calendar dates. Time-of-day is out of scope for this module.

5. **No cascade delete.** Deleting a guest or property that has associated reservations will succeed at the service level but may fail at the DB level once proper FK constraints are added in a migration. This is acceptable while `ddl-auto=update` controls the schema.

## Future Improvements

The following items are out of scope for this implementation but are identified as the logical next steps:

1. **Invoice integration.** When `FacturaService` is built, it will call `ReservationRepository` to read the reservation's `checkIn`, `checkOut`, and `guestCount`, and `PropertyRepository` to read `pricePerNight`. The current module is ready to support this without modification.

2. **Exclude cancelled reservations from overlap check.** A `CANCELLED` reservation currently blocks the dates it held. The repository query should eventually become `...AndStatusNot(ReservationStatus.CANCELLED)` to free those dates for rebooking.

3. **Check-in date must be today or future.** Adding `@FutureOrPresent` to `checkIn` in the DTOs, or a service-level check against `LocalDate.now()`, would prevent creating reservations in the past.

4. **Pagination on `findAll()`.** As the `reserva` table grows, `findAll()` will load unbounded rows into memory. Replacing with `JpaRepository.findAll(Pageable)` should be done before production load.

5. **Filter reservations by property or guest.** Operational queries like "all reservations for property 3" or "all reservations for guest 7" will be needed by the frontend. These are additional `findByPropertyId` / `findByGuestId` derived queries on the repository.

6. **Database FK constraints.** Once the schema is managed by a migration tool (Flyway/Liquibase) rather than `ddl-auto=update`, explicit `FOREIGN KEY (huesped_id) REFERENCES huesped(id)` and `FOREIGN KEY (propiedad_id) REFERENCES propiedad(id)` constraints should be added at the DB level.

7. **Concurrent booking race condition.** Two simultaneous POST requests for the same property and overlapping dates can both pass the overlap check before either write commits. A database-level unique constraint or a pessimistic lock on the property row would eliminate this race. For a class project this is acceptable; for production it must be addressed.
